# Background and Motivation

Проект использует Supabase как облачную базу данных и сервис аутентификации. Основные задачи — расширение функциональности (личные кабинеты, интеграции, улучшение UX) и развитие контент-менеджмента без изменения текущей инфраструктуры.

# Key Challenges and Analysis
- Реализация собственного REST API для всех сущностей (проекты, блог, комментарии, пользователи и т.д.)
- Переписывание сервисов и хуков на фронтенде для работы с новым API
- Организация аутентификации (если требуется)
- Миграция/импорт существующих данных (если нужно)
- Тестирование на соответствие старому функционалу
- Галерея должна работать только для уже созданных проектов (у которых есть id).
- Необходимо реализовать UI для загрузки, удаления, сортировки фото.
- Фото должны храниться в Supabase Storage и быть связаны с проектом через таблицу project_images.
- Нужно обеспечить реактивное обновление галереи после добавления/удаления фото.
- Требуется централизованная система уведомлений через Telegram для всех ролей (клиент, редактор, админ и др.).
- Пользователь может привязать один или несколько Telegram-аккаунтов (особенно клиенты).
- Уведомления о ключевых событиях (workflow, статусы, новые задачи, комментарии и т.д.) должны приходить мгновенно и надёжно.

# Architecture Overview

**Текущие основные компоненты проекта**

1. **Frontend (Vite + React + TypeScript)**  
   • Директория `src/` — страницы (`pages/`), компоненты (`components/`), хуки (`hooks/`), сервисы (`services/`).  
   • Использует Supabase JS-SDK (`src/integrations/supabase`) для аутентификации и CRUD-запросов к Postgres.  
   • Tailwind CSS + shadcn-ui для стилизации, Vite обеспечивает сборку и HMR.

2. **Backend API-сервер (Node.js + Express)**  
   • Главный файл `server.cjs`, вспомогательные модули в `api/`.  
   • Предоставляет REST-эндпоинты для:  
     – генерации Telegram-токенов и управления привязками (`/api/telegram/*`)  
     – отправки уведомлений (`/api/notifications/*`)  
     – приёма данных форм (`/api/forms/submit`) и их отправки в Bitrix24.  
   • Для хранения данных пока использует Supabase Admin API (service key).

3. **Telegram-бот (`aiogram`)**  
   • Файл `ugrabuilders_telegram_bot.py`.  
   • Обрабатывает `/start <token>`, привязывает `telegram_user_id` к `user_id` в таблице `telegram_accounts`, отправляет уведомления.  
   • Запускается из `start-telegram.bat` вместе с API-сервером.

4. **База данных и хранилище**  
   • Supabase Postgres — все таблицы (`projects`, `user_profiles`, `form_submissions`, etc.).  
   • Supabase Storage — изображения проектов / документов.  
   • SQL-скрипты для создания/исправления схемы находятся в корне (`schema.sql`, `create-tables.cjs`, `create-admin-functions.sql`).

5. **Внешние интеграции**  
   • Bitrix24 CRM — web-hook для создания лида из формы.  
   • (Планируется) внутренняя CRM на стороне клиента — сейчас есть заглушка-провайдер.

**Связи и потоки данных**

```
Frontend ─┬─► Supabase (auth + данные)
          └─► API-сервер (формы, Telegram, уведомления)

API-сервер ─► Supabase (запись/чтение)  
API-сервер ─► Bitrix24 (webhook)  
API-сервер ─► Telegram-бот (HTTP-запросы / сообщения)

Telegram-бот ─► Supabase (таблица привязок)  
Telegram-бот ◄─ API-сервер (polling/вебхуки событий)
```

**Ключевые наблюдения / узкие места**

- Зависимость от Supabase критична: фронтенд и бэкенд используют её напрямую.  
- Express-сервер уже существует ⇒ можно постепенно выносить часть бизнес-логики с Supabase Admin API на собственные REST-эндпоинты без изменения фронтенда.  
- Supabase Storage может остаться как object-storage для изображений — миграция не приоритетна.  
- Требуется унифицировать работу с формами и CRM (Bitrix24 + внутренняя система) через сервис интеграции в `src/services/integration`.

## Lessons
- При использовании API Bitrix24 важно учитывать ограничения на количество запросов (rate limit)
- Для отправки данных в Bitrix24 используется метод `crm.lead.add` с различными полями
- Хранение резервной копии данных перед отправкой в CRM может помочь избежать потери заявок при временных проблемах с API
- При интеграции с несколькими системами лучше использовать параллельную отправку данных для ускорения работы
- В клиентском коде нельзя использовать серверные API и переменные окружения (process.env)
- Функция AbortSignal.timeout() поддерживается не во всех браузерах, лучше использовать Promise.race

# Детализация ТЗ: Личный кабинет редактора

## Назначение
Редактор отвечает за визуальное и текстовое наполнение сайта: проекты, фотографии, описания, отзывы, блог. Не имеет доступа к управлению пользователями, финансами, настройками.

## Основные разделы UI
- **Список проектов**: просмотр, фильтрация, поиск, переход к редактированию
- **Сделанные проекты (портфолио)**: отдельный раздел для управления завершёнными/опубликованными проектами, их фото и описаниями
- **Отзывы**: просмотр, модерация, публикация/скрытие отзывов клиентов
- **Блог**: создание, редактирование, публикация статей, загрузка изображений
- **Создание/редактирование проекта**: форма с полями (название, описание, фото, параметры, категория, теги, статус публикации)
- **Медиа-менеджер**: загрузка, удаление, организация фотографий (drag&drop, сортировка, превью)
- **Редактирование текстов**: возможность менять описания, преимущества, заголовки
- **История изменений**: просмотр последних правок (опционально)
- **Уведомления**: сообщения об успехе/ошибке операций

## User Stories
- Как редактор, я вижу список всех проектов и могу быстро найти нужный по фильтру/поиску
- Как редактор, я могу создать новый проект, заполнив все необходимые поля и загрузив фото
- Как редактор, я могу редактировать существующий проект, менять тексты, фото, параметры
- Как редактор, я могу управлять завершёнными проектами (портфолио), их фото и описаниями
- Как редактор, я могу просматривать и модерировать отзывы клиентов, публиковать или скрывать их
- Как редактор, я могу создавать и редактировать статьи блога, загружать изображения
- Как редактор, я могу загружать и удалять фотографии для любого проекта
- Как редактор, я вижу статус публикации проекта, статьи, отзыва и могу его менять
- Как редактор, я получаю уведомления об успехе/ошибке операций
- Как редактор, я не вижу разделы, связанные с пользователями, финансами, настройками

## Основные сценарии
1. Просмотр и поиск проектов
2. Создание нового проекта
3. Редактирование существующего проекта
4. Управление завершёнными проектами (портфолио)
5. Загрузка/удаление фотографий
6. Изменение текстов и параметров
7. Публикация/снятие с публикации проектов
8. Просмотр, модерация и публикация отзывов
9. Создание, редактирование и публикация статей блога

## API/Backend
- GET /projects — список проектов (с фильтрами)
- POST /projects — создать проект
- PUT /projects/:id — обновить проект
- DELETE /projects/:id — удалить проект (опционально)
- POST /projects/:id/images — загрузить фото
- DELETE /projects/:id/images/:imageId — удалить фото
- PUT /projects/:id/publish — сменить статус публикации
- GET /portfolio — список завершённых проектов
- PUT /portfolio/:id — обновить портфолио-проект
- GET /reviews — список отзывов
- PUT /reviews/:id/publish — опубликовать/скрыть отзыв
- GET /blog — список статей
- POST /blog — создать статью
- PUT /blog/:id — обновить статью
- DELETE /blog/:id — удалить статью
- POST /blog/:id/images — загрузить изображение для статьи
- PUT /blog/:id/publish — сменить статус публикации статьи

## Критерии готовности
- Редактор может полностью управлять контентом проектов, портфолио, отзывов и блога (CRUD, фото, тексты)
- Нет доступа к управлению пользователями, финансами, настройкам
- Все основные сценарии протестированы (UI, API, уведомления)

## 2. Workflow подтверждения
- Любое новое или изменённое содержимое (проект, отзыв, статья) создаётся в статусе "черновик"
- Редактор отправляет на подтверждение (статус "на подтверждении")
- Администратор видит все заявки на подтверждение в отдельном разделе ("Ожидают подтверждения")
- Админ может:
  - Одобрить (статус "опубликовано")
  - Отклонить (статус "отклонено", с комментарием)
- После одобрения контент становится видимым для всех пользователей
- После отклонения редактор видит причину и может доработать

## 3. Компоненты и структура
- `ContentDashboard` — корневой компонент панели редактора/админа
- `ProjectListTable` — таблица проектов с фильтрами и статусами
- `ProjectRow` — строка проекта в таблице
- `StatusBadge` — компонент для отображения статуса (черновик, на подтверждении, опубликовано, отклонено)
- `ActionMenu` — меню действий для проекта (редактировать, отправить на подтверждение, удалить)
- `PortfolioList` — список завершённых проектов
- `PortfolioProjectCard` — карточка проекта в портфолио
- `ReviewListTable` — таблица отзывов с модерацией
- `ReviewRow` — строка отзыва в таблице
- `BlogListTable` — список статей блога
- `BlogRow` — строка статьи в списке
- `AdminApprovalPanel` — раздел для подтверждения заявок (только для админа)
- `ApprovalRequestCard` — карточка заявки на подтверждение
- `ApproveButton` — кнопка одобрения заявки
- `RejectButton` — кнопка отклонения заявки
- `CommentBox` — поле для комментария при отклонении
- `ProjectEditForm` — форма создания/редактирования проекта
- `BlogEditForm` — форма создания/редактирования статьи
- `MediaManager` — загрузка/организация фото
- `HistoryPanel` — история изменений (опционально)

## 4. UI/UX детали
- Для редактора: доступны только свои черновики, отправка на подтверждение, просмотр статусов
- Для админа: доступны все проекты/отзывы/статьи, раздел "Ожидают подтверждения", кнопки "Одобрить/Отклонить"
- Все действия сопровождаются уведомлениями (успех/ошибка)
- Везде, где требуется подтверждение — отображается статус и история

## 5. Пример workflow (проект)
1. Редактор создаёт проект → статус "черновик"
2. Редактор отправляет на подтверждение → статус "на подтверждении"
3. Админ видит заявку, одобряет → статус "опубликовано" (или отклоняет с комментарием)
4. Редактор видит результат, может доработать и повторно отправить

## 6. Критерии готовности
- Все разделы работают согласно ролям и workflow
- Админ может подтверждать/отклонять любые заявки
- Существующий функционал не нарушен
- Все сценарии протестированы (UI, статусы, уведомления, права)

# Детализация ТЗ: Личный кабинет клиента

## Назначение
Личный кабинет клиента – центральная точка взаимодействия заказчика с компанией, позволяющая отслеживать ход строительства, получать документы и общаться со специалистами. Интерфейс должен быть максимально понятным и информативным, с акцентом на визуальную составляющую.

## Анализ текущего состояния
В системе уже реализованы:
- Базовая аутентификация пользователей
- Хранение данных о пользователях в Supabase
- Привязка Telegram для уведомлений
- Возможность просмотра файлов (документы, фото)

Для полноценного личного кабинета необходимо расширить функциональность.

## Ключевые компоненты и функции

### 1. Дашборд клиента
- **Сводка по проекту**: название, адрес, тип дома, основные характеристики
- **Текущий статус строительства**: визуальный прогресс-бар с этапами
- **Текущий этап продажи**: визуальный индикатор стадии сделки
- **Ближайшие события**: календарь с датами важных событий
- **Уведомления**: счетчик и список непрочитанных уведомлений

### 2. Строительный прогресс
- **Визуальный таймлайн** с этапами строительства:
  - Подготовка участка
  - Фундамент
  - Коробка
  - Крыша
  - Инженерные системы
  - Отделка
  - Финальные работы
- **Текущая стадия** должна быть выделена
- **Процент завершения** общий и для каждого этапа
- **Прогнозируемые даты** завершения этапов

### 3. Фотоотчеты
- **Галерея фотографий** по этапам строительства
- **Фильтрация фото** по датам и этапам
- **Просмотр фото** в полноэкранном режиме с описаниями
- **Хронологическая лента** с последними фото
- **Уведомления** о новых фотоотчетах

### 4. Документы
- **Категории документов**:
  - Договоры и приложения
  - Технические документы
  - Финансовые документы
  - Акты и отчеты
- **Просмотр документов** онлайн и возможность скачивания
- **Статус документов**: на подписании, подписан, требует внимания
- **Уведомления** о новых документах

### 5. Контакты и коммуникации
- **Контактная информация** ответственных лиц:
  - Менеджер проекта (имя, фото, телефон, email)
  - Бригадир (имя, фото, телефон)
  - Техническая поддержка
- **Возможность быстрой связи**: звонок, email, чат
- **История коммуникаций**: лог сообщений и звонков
- **Форма обратной связи** для вопросов и пожеланий

### 6. Управление уведомлениями
- **Настройка каналов** получения уведомлений:
  - В личном кабинете
  - Email
  - Telegram
  - SMS (опционально)
- **Настройка типов уведомлений**:
  - Изменение статуса строительства
  - Новые фотоотчеты
  - Новые документы
  - Сообщения от менеджера/бригадира
  - Финансовые операции

### 7. Финансовый раздел (опционально, при необходимости)
- **График платежей** с датами и суммами
- **История платежей** с квитанциями
- **Предстоящие платежи** с возможностью онлайн-оплаты
- **Сводка по финансам** проекта

## User Stories

1. **Основной сценарий**
   - Как клиент, я хочу при входе в личный кабинет сразу видеть текущий статус строительства и последние обновления
   - Как клиент, я хочу просматривать фотографии строительства в хронологическом порядке
   - Как клиент, я хочу получать и просматривать документы, связанные с моим проектом
   - Как клиент, я хочу иметь быстрый доступ к контактам менеджера и бригадира

2. **Прогресс строительства**
   - Как клиент, я хочу видеть детальный прогресс строительства с разбивкой по этапам
   - Как клиент, я хочу получать уведомления о переходе на новый этап строительства
   - Как клиент, я хочу видеть прогнозируемые даты завершения текущего этапа и всего строительства

3. **Фотоотчеты**
   - Как клиент, я хочу получать регулярные фотоотчеты с площадки
   - Как клиент, я хочу фильтровать фотографии по датам и этапам
   - Как клиент, я хочу просматривать фотографии в полноэкранном режиме с возможностью увеличения

4. **Документооборот**
   - Как клиент, я хочу видеть все свои документы в одном месте
   - Как клиент, я хочу получать уведомления о новых документах, требующих внимания
   - Как клиент, я хочу скачивать документы для офлайн-доступа

5. **Коммуникации**
   - Как клиент, я хочу иметь возможность быстро связаться с менеджером
   - Как клиент, я хочу оставлять вопросы и пожелания по проекту
   - Как клиент, я хочу получать ответы на свои вопросы в личном кабинете

6. **Уведомления**
   - Как клиент, я хочу настраивать способы получения уведомлений
   - Как клиент, я хочу видеть историю всех уведомлений
   - Как клиент, я хочу интегрировать уведомления с Telegram

## Lessons
- При использовании API Bitrix24 важно учитывать ограничения на количество запросов (rate limit)
- Для отправки данных в Bitrix24 используется метод `crm.lead.add` с различными полями
- Хранение резервной копии данных перед отправкой в CRM может помочь избежать потери заявок при временных проблемах с API
- При интеграции с несколькими системами лучше использовать параллельную отправку данных для ускорения работы
- В клиентском коде нельзя использовать серверные API и переменные окружения (process.env)
- Функция AbortSignal.timeout() поддерживается не во всех браузерах, лучше использовать Promise.race
